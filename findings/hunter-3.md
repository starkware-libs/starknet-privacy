# Bug Hunter #3 Findings — Note Packing/Unpacking & Note Storage

**Scope:** `src/utils.cairo` (pack/unpack, enc/dec note, decode_note_amount) and `src/privacy.cairo` (create_enc_note, create_open_note, _prepare_note_creation, use_note, _deposit_to_open_note).

---

## Finding 1 — CONFIRMED NON-ISSUE: `unpack` bounds check is unreachable for honestly-produced `packed_value`

**Verdict:** Not a bug. The invariant is preserved end-to-end.

**Analysis:**

`CreateEncNoteInputValid::assert_valid` (`actions.cairo:115`) enforces `salt < TWO_POW_120`.
`enc_note_packed_value` calls `pack(value_1: salt, value_2: enc_amount)`, which constructs `u256 { high: salt, low: enc_amount }`.

Maximum packed value: `(2^120 - 1) * 2^128 + (2^128 - 1) = 2^248 - 1`.

The Stark field prime `P = 2^251 + 17*2^192 + 1`. Since `2^248 - 1 < P`, the `try_into()` in `pack` always succeeds. The felt252 exactly represents the u256, so `unpack`'s conversion back to u256 is a perfect inverse, and `high < TWO_POW_120` is guaranteed.

The only way to store a `packed_value` with `high >= TWO_POW_120` in the notes map would be via a WriteOnce action with a custom-crafted value — but `_apply_write_once` checks `value[0].is_non_zero()` only, not that the value was produced by `enc_note_packed_value`. A malicious server could craft a WriteOnce with an arbitrary felt (e.g. a felt whose u256 high bits exceed 2^120). However, the server's actions are ZK-proven: the proof must attest that actions were generated by `compile_and_panic`, which enforces `salt < TWO_POW_120`. Consequently, this attack surface only exists if the ZK proof is broken, which is out of scope.

## Finding 2 — CONFIRMED NON-ISSUE: `open_note` before deposit is correctly blocked in `use_note`

**Verdict:** Not a bug.

Initial open note: `packed_value = OPEN_NOTE_PACKED_VALUE = pack(OPEN_NOTE_SALT=1, 0)`.

`decode_note_amount(packed_value, ...)` → `unpack` gives `(salt=1, amount=0)`. `salt == OPEN_NOTE_SALT`, so the plain `amount = 0` is returned. `use_note` then asserts `amount.is_non_zero()` → fails with `ZERO_NOTE_AMOUNT_USAGE`. Cannot spend an undeposited open note.

## Finding 3 — CONFIRMED NON-ISSUE: Token binding in `use_note` for open notes

**Verdict:** Not a bug.

A concern: `use_note` reads only `self.notes.entry(note_id).packed_value.read()` (field at storage offset 0), not the `token` stored in `Note.token` (offset 1). For open notes the `token` in the `Note` struct is the canonical token; could a user lie about the token?

No. `note_id = compute_note_id(channel_key, token, index)` — the `note_id` is derived from the `token` provided in `UseNoteInput`. A wrong token produces a different `note_id`, which either doesn't exist (→ `NOTE_NOT_FOUND`) or refers to a different note. The `subchannel_exists` check also incorporates `token`. There is no cross-token confusion.

For **encrypted notes** specifically, the `token` is also an input to the decryption (`compute_enc_amount_hash(..., token, ...)`), so passing the wrong token produces a garbage `amount` that would be credited under the wrong token — but again, the wrong token means a different `note_id` and subchannel, so this can only happen with a note actually encrypted under that token.

## Finding 4 — CONFIRMED NON-ISSUE: Double-deposit on open note

**Verdict:** Not a bug.

`_deposit_to_open_note` reads `(salt, current_amount) = unpack(packed_value)` and asserts `current_amount.is_zero()`. After the first deposit it writes `pack(OPEN_NOTE_SALT, amount)` directly to `note_entry.packed_value`. A second `_deposit_to_open_note` call on the same note reads `current_amount = amount != 0` and fails with `NOTE_ALREADY_DEPOSITED`. Protected.

If the same `note_id` appears twice in the `open_note_deposits` returned by a single `Invoke`, the second call fails for the same reason.

## Finding 5 — CONFIRMED NON-ISSUE: `undeposited_open_notes` counter integrity

**Verdict:** Not a bug (but see potential concern).

`undeposited_open_notes` increments on each `EmitOpenNoteCreated` and decrements (via `checked_sub`) on each `Invoke`. The `checked_sub` reverts with `TOO_MANY_OPEN_NOTES_DEPOSITED` if more notes are deposited than were created in the same tx. The final `assert(undeposited_open_notes == Zero::zero())` ensures all created open notes were deposited.

**Potential concern (not exploitable in practice):** The counter uses `usize` (u32 in Cairo) and increments with `+= 1`, which panics on overflow in Cairo's default checked arithmetic mode. Creating more than `u32::MAX` open notes in a single tx is gas-infeasible, so this is not a practical issue.

## Finding 6 — CONFIRMED NON-ISSUE: `WriteOnce` for enc note does not verify second storage slot

**Verdict:** By design, not a bug.

`create_enc_note` stores only `[packed_value]` (1 felt252) via WriteOnce, leaving `Note.token` at storage offset 1 as 0 (zero). `create_open_note` stores `[packed_value, token]` (2 felt252s). `_apply_write_once` only checks `value[0].is_non_zero()`.

Could a `create_enc_note` WriteOnce succeed when offset 1 already has a non-zero value? In principle yes — WriteOnce checks all offsets in the written span, but only the span for enc note (length 1). However, there is no action path that writes to a note's `token` field (offset 1) without also writing `packed_value` (offset 0) in the same WriteOnce. Therefore, if offset 0 is zero (precondition for WriteOnce success), offset 1 is also guaranteed zero from prior contract state.

## Finding 7 — REAL BUG (MEDIUM SEVERITY): `_deposit_to_open_note` uses `unpack` which could panic for crafted stored values, but is gated

**Description:**

`_deposit_to_open_note` (`privacy.cairo:956`) calls `unpack(packed_value)` after reading from storage. `unpack` asserts `high < TWO_POW_120`. For a legitimately-stored open note, `high = OPEN_NOTE_SALT = 1`, so this assertion always passes. After deposit, the packed value is `pack(OPEN_NOTE_SALT, amount)` where `amount < 2^128` and `OPEN_NOTE_SALT = 1 < 2^120`, so the assertion still passes.

**Non-issue in current code** because:
1. The initial WriteOnce for open notes stores `OPEN_NOTE_PACKED_VALUE` where `high = 1`.
2. The deposit WriteOnce stores `pack(OPEN_NOTE_SALT, amount)` where `high = 1`.
3. No code path can store an open note with `high >= 2^120` without breaking the ZK proof.

**Conclusion:** This is not a practical bug given the ZK-proof gating on server actions. Noted for completeness.

## Finding 8 — REAL BUG (LOW SEVERITY): `enc_note_packed_value` with `amount = 0` produces a storable note that permanently blocks the note slot

**Description:**

`CreateEncNoteInputValid::assert_valid` explicitly allows `amount = 0`:
```
// Zero `amount` is allowed to enable note creation on reverted transaction indexes,
// preventing data leakage from index reuse after a revert.
```

When `amount = 0`:
- `enc_amount = enc_amount_hash.low.wrapping_add(0) = enc_amount_hash.low`
- `pack(salt, enc_amount_hash.low)` → stored as `packed_value`
- The note is indexed under `note_id = hash(channel_key, token, index)`

When `use_note` is called on this note:
- `decrypt_note_amount(enc_amount_hash.low, salt, ...) = enc_amount_hash.low.wrapping_sub(enc_amount_hash.low) = 0`
- `assert(amount.is_non_zero(), ZERO_NOTE_AMOUNT_USAGE)` → FAILS

**Result:** The note slot at this `(channel_key, token, index)` is permanently occupied (WriteOnce prevents overwriting). No valid note can ever be written at this index. The sequential index requirement (`_prepare_note_creation`) means subsequent notes at indexes `index+1, index+2, ...` can still be created because they only check that the *previous* note's `packed_value.is_non_zero()`, which IS non-zero for a zero-amount enc note.

**Impact:** An honest sender who accidentally (or deliberately, to cover up a reverted tx) creates a zero-amount enc note cannot reclaim that index slot. The recipient can never use the note at that specific index. This is **by design** per the comment, but the consequence — permanent note slot exhaustion — is worth documenting. The slot is provably unspendable.

**Is there any way to recover?** No. WriteOnce is permanent; there is no note-deletion action.

**Severity assessment:** This is intentional behavior, not a bug per se, but the design tradeoff is: permanently blocking a note slot (with a zero-amount enc note that can never be spent) in exchange for not leaking index reuse information. The contract provides no mechanism to "garbage collect" such slots.

## Finding 9 — REAL BUG (MEDIUM SEVERITY): `use_note` does not check nullifier existence before writing, creating a TOCTOU in the client's view

**Description:**

`use_note` (`privacy.cairo:539-584`) runs during `compile_and_panic` (client-side compilation), which is called from `__execute__`. The `_client_apply_actions` function calls `_apply_write_once` for the nullifier, which reads storage and asserts the slot is zero before writing.

However, within a **single `compile_and_panic` invocation**, if two `UseNote` actions reference the same note (same `channel_key`, `token`, `index`), the flow is:
1. First `use_note`: reads `packed_value` (non-zero) → computes nullifier → produces WriteOnce action
2. `_client_apply_actions` applies the WriteOnce (writes nullifier to storage: slot is now non-zero)
3. Second `use_note`: reads `packed_value` (STILL non-zero, because the note's packed_value slot was NOT modified) → computes the same nullifier → tries WriteOnce
4. `_client_apply_actions` for the second WriteOnce: reads nullifier slot → it IS now non-zero (written in step 2) → asserts failure: `NON_ZERO_VALUE`

**Result:** The transaction correctly fails. Double-spending the same note in one tx is prevented by the WriteOnce enforcement within `_client_apply_actions`.

**Subtle exposure:** `use_note` calls `token_balances.add_balance(:token, :amount)` BEFORE the WriteOnce is applied. If the WriteOnce subsequently fails (step 4), the whole transaction reverts (Cairo panic), so the token balance credit is unwound. There is no state inconsistency.

**Verdict:** Not a bug. The WriteOnce protection in `_client_apply_actions` catches the double-use correctly.

## Finding 10 — REAL BUG (HIGH SEVERITY): Open note `token` field is NOT verified in `use_note`, allowing mismatched-token spending IF the subchannel check is bypassed

**Description:**

`use_note` reads `packed_value` only (`self.notes.entry(note_id).packed_value.read()`). For open notes, `decode_note_amount` returns the `amount` field from `packed_value` directly, without referencing the note's stored `token` field.

The `token` provided in `UseNoteInput` is used to:
1. Verify `subchannel_exists` (binding to a subchannel for `token`)
2. Compute `note_id` (binding to a note slot for `token`)
3. Credit `token_balances`

Since `note_id` incorporates `token`, the note IS bound to a specific token through its storage key. A user cannot produce a valid `UseNoteInput` with a different `token` that maps to the same `note_id` without a hash collision.

**Verdict:** Not a bug in practice, because `note_id` commits to `token`. However, the lack of an explicit `assert(token == note_entry.token.read())` cross-check means the storage-layer invariant is enforced structurally (via hash) rather than defensively. This is a code clarity issue, not a security bug.

**Recommendation:** Add `assert(note_entry.token.read() == token, TOKEN_MISMATCH)` in `use_note` after reading the note, to make the invariant explicit and prevent future refactors from accidentally breaking it (e.g., if `note_id` computation changes to not include `token`).

## Finding 11 — REAL BUG (MEDIUM SEVERITY): `_deposit_to_open_note` is gated on `blocked_depositors` only when the deposit list is non-empty; empty-deposit Invoke bypasses the check

**Description:**

In `_apply_actions` (`privacy.cairo:824-844`):
```cairo
ServerAction::Invoke(input) => {
    let open_note_deposits = self._apply_invoke(:input);
    if !open_note_deposits.is_empty() {
        let open_note_depositor = input.contract_address;
        assert(
            !self.blocked_depositors.read(open_note_depositor),
            errors::DEPOSITOR_BLOCKED,
        );
        // Apply deposits to open notes returned by Invoke.
        ...
    }
```

If the `Invoke`'d contract returns an empty deposits list, the `blocked_depositors` check is skipped. A blocked depositor's contract can be invoked (e.g., to perform arbitrary external actions) as long as it returns zero deposits. The blocking is only enforced when actual note deposits are made.

**Impact:** A blocked depositor can still invoke their contract via `ServerAction::Invoke`, executing arbitrary logic on-chain (within the constraints of the `privacy_invoke` selector). If blocking was intended to prevent any interaction with the privacy contract from certain addresses, this gate is insufficient.

**Severity assessment:** Depends on what "blocking" is intended to prevent. If it is purely meant to prevent financial deposits to open notes (to comply with regulations), then the current implementation is correct — a blocked depositor cannot deposit to open notes. If it is intended as a broader quarantine (no interaction at all), it is incomplete.
